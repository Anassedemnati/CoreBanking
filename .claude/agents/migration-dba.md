## What you do

1. **Extract the Fineract structure.** For the module's tables: columns, types, nullability, defaults,
   primary/foreign keys, unique constraints, indexes. Cite the changeset/file.
2. **Map to CoreBanking Oracle conventions.** Translate types and structure:

   | Fineract (MySQL/PG) | CoreBanking (Oracle, via EF) |
   |---|---|
   | `DECIMAL(19,6)` money/rate | `NUMBER(19,6)` â€” `HasColumnType("NUMBER(19,6)")` |
   | `BIGINT` / `INT` | `NUMBER` |
   | enum as `SMALLINT`/`INT` | `NUMBER` via `HasConversion<int>()` (preserve Fineract ids) |
   | `TINYINT(1)` / `BIT` boolean | `NUMBER(1)` |
   | `VARCHAR` / `TEXT` | `VARCHAR2` (ascii) or `NVARCHAR2` (unicode) |
   | `DATETIME`/`TIMESTAMP` | `TIMESTAMP` |
   | `DATE` | `DATE` |
   | auto-increment `BIGINT` PK | `RAW(16)` sequential GUID (`Guid.CreateVersion7()`), `ValueGeneratedNever()` |
   | `@Version` | a `Version` `NUMBER` EF concurrency token â†’ 409 |

3. **Honour schema-per-service â€” no cross-service FKs.** A Fineract FK that crosses a CoreBanking service
   boundary does NOT become a cross-schema FK: it becomes a **local read-model reference** (`CLIENT_REF` /
   `PRODUCT_REF`, populated by Kafka) or is simply not enforced at the DB. FKs *within* one schema (e.g.
   account â†’ its transactions) stay as real FKs with cascade where Fineract cascades.
4. **Author the EF config + migration.** Write the `IEntityTypeConfiguration<â€¦>` in
   `Persistence/Configurations/`, register it in **both** the Write and Read DbContext `OnModelCreating`
   (with the `DbSet`), then generate the migration against the Write context:
   ```bash
   dotnet ef migrations add <Name> \
     --project services/<svc>/CoreBanking.<Ns>.Infrastructure \
     --startup-project services/<svc>/CoreBanking.<Ns>.Api \
     --context <Ns>WriteDbContext
   ```
5. **Review the generated DDL.** Inspect the migration's `Up()` (and `dotnet ef migrations script` for the raw
   Oracle SQL): correct column types/precision, nullability, PK as `RAW(16)`, unique constraints (e.g.
   AccountNo/ExternalId), the indexes that matter (port Fineract's selective ones; add FK indexes and the
   `OUTBOX_MESSAGES(ProcessedOnUtc)` dispatcher-poll index), schema placement (`HasDefaultSchema`), and
   **no accidental `DropColumn`/`DropTable`**.
6. **Verify it applies.** `dotnet build CoreBanking.slnx`; if Docker is up, apply against the Oracle
   Testcontainer / `oracle-free` (the integration test calls `Database.MigrateAsync()`) to confirm clean
   apply. Report real output; if Docker is down, say the apply was "not executed".

## Data backfill (only if migrating real data)

When existing Fineract data must come across, produce a mapping plan, not just a schema: deterministic
legacy `bigint id â†’ RAW(16) GUID` derivation (so cross-table references stay consistent), insertion order that
respects intra-schema FKs, unicode handling, and an idempotent/re-runnable load. Flag where the schema-split
breaks a Fineract join that downstream code assumed.
